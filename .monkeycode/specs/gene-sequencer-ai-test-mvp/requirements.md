# 基因测序仪 AI 测试条目与验证方案生成工具 MVP 需求规格说明书

## 1. 产品定位

本系统面向基因测序仪产品测试工程师，基于统一资料池、测试资产库、风险知识源和大模型能力，辅助分析新的开发需求和变更需求，生成测试条目，并在用户审核后按既有验证方案模板导出 Word 验证方案。

MVP 覆盖算法和软件之外的仪器子系统、物料级测试、子系统测试、系统级功能性能测试，并支持多个项目工作空间。

## 2. MVP 目标

MVP 需要跑通以下闭环：

```text
用户上传资料
→ 系统智能识别标签
→ 用户补充缺失标签
→ 系统判断重复资料
→ 管理员审核发布
→ 系统拆分历史验证方案
→ 生成测试条目资产和测试归口包
→ 系统解析 Jira 和 DFMEA 风险资料
→ 用户输入新需求或变更需求
→ 系统推荐测试条目
→ 用户审核测试条目
→ 系统生成验证方案草稿
→ 导出 Word 验证方案
```

## 3. 用户角色

| 角色 | 核心职责 | MVP 权限 |
|---|---|---|
| 系统管理员 | 维护系统基础数据和资料质量 | 用户管理、项目管理、资料审核、标签体系、模板管理、联网开关 |
| 项目管理员 | 管理项目范围和成员 | 项目成员管理、项目资料范围配置、项目生成记录查看 |
| 测试工程师 | 贡献资料并生成验证方案 | 上传资料、补充标签、输入需求、审核推荐条目、生成方案 |
| 审核人员 | 审核测试内容 | 审核测试条目、审核方案草稿、查看风险覆盖 |

权限采用 RBAC、项目 ACL 和资料标签过滤组合控制。

## 4. 资料组织方式

系统采用统一资料池，所有资料只上传一次，通过标签、适用范围、权限和项目引用规则实现复用与隔离。

资料类型包括：

| 类型 | 示例 |
|---|---|
| 产品资料 | 产品说明、系统架构、模块说明 |
| 开发需求 | 新功能需求、设计输入 |
| 变更需求 | ECR、ECN、变更说明 |
| 技术要求 | 产品技术规格书、部件规格书 |
| 测试规范 | 测试标准、测试方法、验收规则 |
| 验证方案 | 历史验证方案 |
| 测试报告 | 历史验证报告 |
| Jira 问题 | 缺陷、风险、验证问题 |
| DFMEA | 失效模式和影响分析 |

资料状态包括：待解析、待补充标签、待去重确认、待管理员审核、已发布、已驳回、已废止。

## 5. 资料上传与审核需求

用户上传资料后，系统应自动解析资料内容并识别标签。未识别或低置信度标签由用户补充或确认。系统应执行重复资料判断，管理员审核通过后资料进入已发布状态并参与检索。

必填标签包括：文档名称、文档类型、产品型号、对象或测试对象、版本、保密级别、适用范围。

系统推荐标签包括：产品线、一级系统域、二级子系统、测试层级、变更类型、关键词、文档摘要。

重复资料判断分为文件级、内容级和语义级。完全重复资料应阻止重复发布，高度相似资料应进入管理员确认。

## 7. 测试方案拆分需求

系统应对验证方案类文档进行条目级拆分。一个 `3.x` 测试项目默认生成一个测试条目资产。系统应提取测试目的、测试标准、方法原理、测试工具、测试步骤、连接图或照片、测试记录、需求符合性和 BUG 信息。

拆分后的每个测试条目应分别映射测试对象、测试层级、测试类型、主子系统、关联子系统和分类依据。

## 8. 测试条目资产需求

测试条目资产是可复用测试知识单元。它既可来自历史方案拆分，也可来自 AI 生成和用户审核。

测试条目应包含：名称、来源文档、来源章节、对象、主子系统、关联子系统、测试层级、测试类型、目的、标准、方法、工具、步骤、记录模板、样本量、预估用时、依据来源、风险覆盖、审核状态。

## 9. 测试归口资产需求

测试归口资产用于表达某类对象或某类变更应组合哪些测试条目。

示例：

```text
RFID 供应商变更验证包
- RFID在机装配测试：必测
- RFID装配后初始化测试：必测
- RFID在机读取测试：必测
- RFID在机写入测试：建议
- 安规EMC测试：条件触发
```

系统应在拆分历史验证方案时生成候选归口包，用户确认后由管理员审核发布。新需求分析时，系统应根据对象和变更类型匹配归口包并推荐包内测试条目。

## 10. 风险知识源需求

Jira 和 DFMEA 应作为风险驱动测试生成的重要资料。

Jira 主要用于历史问题回归、防复发和风险补充。DFMEA 主要用于潜在失效模式、失效后果、失效原因、控制措施和 RPN/AP 驱动的测试建议。

系统应将 Jira、DFMEA、风险管理表、测试报告问题解析为统一风险项，并在新需求分析时检索相关风险项，生成风险补充测试建议。

## 11. 新需求与变更需求分析需求

用户输入新需求或变更需求后，系统应识别变更对象、变更类型、影响产品型号、影响子系统、测试层级、风险点和缺失信息。

系统应匹配测试归口包、召回测试条目资产、检索本地知识库和风险知识源，并输出必测、建议、条件触发、Jira 风险补充、DFMEA 风险补充、AI 补充等分组推荐。

用户应能采纳、修改、删除、补充、合并、拆分测试条目，并对条件触发项做纳入、排除或待确认决策。

## 12. 联网大模型辅助需求

系统可支持联网大模型作为外部参考，但默认关闭，由管理员按项目或组织开启。

联网能力只用于通用测试方法、法规标准解释、行业经验参考和兜底建议。外部参考必须标记来源，且不能覆盖内部受控资料。涉及内部需求、供应商、项目编号、产品参数时，系统应先脱敏再联网。

## 13. 验证方案模板 V1

验证方案模板采用用户提供的 Word 示例结构：

```text
文档履历

1. 概述
   1.1 验证的背景、目的和范围
   1.2 DUT描述
   1.3 参考文档

2. 测试项目列表

3. 测试项目
   3.x 测试项目名称
      3.x.1 测试目的/测试标准
      3.x.2 测试方法/原理
      3.x.3 测试工具
      3.x.4 测试步骤
      3.x.5 测试连接图或照片
      3.x.6 测试记录
      3.x.7 需求符合性和BUG信息
```

## 14. 验证方案生成需求

系统应将用户审核后的测试条目生成结构化验证方案草稿。用户可编辑文档履历、概述、DUT、参考文档、测试项目列表、测试项目详情、测试记录、需求符合性和 BUG 信息。

导出前系统应进行完整性校验。校验通过后，系统应使用固定 Word 模板导出 `.docx` 文件，并保留导出记录和版本追溯。

## 15. MVP 验收标准

| 编号 | 标准 |
|---|---|
| A1 | 支持多用户、多项目和角色权限 |
| A2 | 支持统一资料池上传、标签识别、去重和审核 |
| A3 | 验证方案可拆分为测试条目资产 |
| A4 | 可生成测试归口包并建立测试集合关联 |
| A5 | Jira 和 DFMEA 可解析为风险知识源 |
| A6 | 输入新需求后可推荐测试条目 |
| A7 | 推荐条目展示依据来源和推荐理由 |
| A8 | 用户可审核和修改测试条目 |
| A9 | 可按模板生成验证方案草稿 |
| A10 | 可导出符合模板结构的 Word 文件 |
