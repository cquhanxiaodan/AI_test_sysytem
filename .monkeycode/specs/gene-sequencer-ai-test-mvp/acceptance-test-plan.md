# MVP 验收测试方案与评估指标

## 1. 验收目标

MVP 验收需要验证流程可跑通、结果可用、依据可追溯、用户可控制。

| 目标 | 说明 |
|---|---|
| 流程可跑通 | 从资料上传到 Word 导出完整闭环 |
| 结果可用 | 生成测试条目对测试工程师有参考价值 |
| 依据可追溯 | 测试条目能看到来源资料、DFMEA、Jira 或用户确认 |
| 用户可控制 | 标签、条目、归口包和方案都能人工确认和修改 |

## 2. 验收数据集

| 数据类型 | 数量 | 要求 |
|---|---:|---|
| 验证方案 Word | 5-10 份 | 覆盖 RFID、温控、流体、光学、电气等对象 |
| 测试报告 | 3-5 份 | 包含测试结论、问题、异常、数据 |
| DFMEA | 1-3 份 | 包含失效模式、后果、原因、S/O/D/RPN/AP |
| Jira 导出表 | 1-3 份 | 包含问题描述、根因、复现步骤、状态、RPN |
| 测试规范 | 3-5 份 | 包含测试方法、判定标准 |
| 技术要求 | 3-5 份 | 包含产品指标、部件规格 |
| 新需求/变更需求 | 5-10 条 | 覆盖供应商变更、型号变更、结构变更、参数变更 |

RFID 示例必须作为端到端主验收案例。

## 3. 端到端主验收用例

### TC-E2E-001：RFID 供应商变更验证方案生成

前置条件：系统已有用户、项目 DNBSEQ-G99、验证方案模板 V1。

输入资料：`DNBSEQ-G99 ECR4.1康奈特RFID验证方案.docx`。

输入需求：`G99 引入新的 RFID 二供供应商，需要评估验证测试。`

测试步骤：

1. 测试工程师上传 RFID 验证方案。
2. 系统解析文档并识别标签。
3. 用户确认标签并提交审核。
4. 管理员审核发布资料。
5. 系统拆分测试条目。
6. 系统生成 RFID 供应商变更验证包。
7. 用户输入新的 RFID 供应商变更需求。
8. 系统识别变更对象和变更类型。
9. 系统匹配 RFID 供应商变更验证包。
10. 系统推荐测试条目。
11. 用户审核测试条目。
12. 系统生成验证方案草稿。
13. 用户执行完整性校验。
14. 系统导出 Word。

验收标准：

| 编号 | 标准 |
|---|---|
| E2E-A1 | 系统识别文档类型为验证方案 |
| E2E-A2 | 系统识别产品型号为 DNBSEQ-G99 |
| E2E-A3 | 系统识别测试对象为 RFID |
| E2E-A4 | 系统拆分出 5 个测试条目 |
| E2E-A5 | 系统生成 RFID 供应商变更验证包 |
| E2E-A6 | 需求分析识别变更对象 RFID 和供应商变更 |
| E2E-A7 | 推荐装配、初始化、读取、写入、安规 EMC 测试 |
| E2E-A8 | 每个推荐项显示推荐理由和依据 |
| E2E-A9 | 用户可修改和删除测试条目 |
| E2E-A10 | 导出 Word 结构符合模板 V1 |

## 4. 资料上传与标签识别测试

### TC-DOC-001：验证方案标签识别

输入 RFID 验证方案 Word，系统应识别文档类型、产品型号、测试对象、变更类型、版本、一级系统域和二级子系统。

### TC-DOC-002：低置信度标签用户补充

系统应标记未识别或低置信度标签，用户补齐后可提交审核。

### TC-DOC-003：重复资料识别

重复上传同一份资料时，系统应通过文件 hash 识别完全重复并提示已有资料。

### TC-DOC-004：高度相似资料识别

同一方案轻微修改版本应被识别为高度相似，管理员可选择合并为新版本。

## 5. 测试方案拆分测试

### TC-SPLIT-001：RFID 方案拆分测试条目

系统应从 RFID 验证方案拆分出：RFID在机装配测试、RFID装配后初始化测试、RFID在机读取测试、RFID在机写入测试、安规EMC测试。

### TC-SPLIT-002：条目级子系统映射

每个条目应有主子系统和关联子系统。例如 RFID 在机读取测试主分类为试剂盒接口子系统，关联信号采集子系统和整机性能子系统。

### TC-SPLIT-003：测试归口包生成

系统应生成 RFID 供应商变更验证包，并将 5 个测试条目挂入包内，标记必测、建议或条件触发。

## 6. 风险知识源测试

### TC-RISK-001：DFMEA 解析

系统应将 DFMEA 有效行解析为风险项，识别 DFMEA 编号、失效模式、失效后果、失效原因、S/O/D/RPN/AP。

### TC-RISK-002：Jira 表单解析

系统应将 Jira 导出表解析为风险项，识别 Jira 编号、问题描述、复现步骤、根因、状态和 RPN。

### TC-RISK-003：风险补充测试建议

当新需求命中相关风险项时，系统应生成风险驱动测试建议，并标记 Jira 或 DFMEA 来源。

## 7. 需求分析与推荐测试

### TC-REQ-001：RFID 供应商变更需求解析

输入：`DNBSEQ-G99 由于物料供应原因，需要引入新的 RFID 二供供应商，要求评估该 RFID 在整机上的适配性和使用性能。`

预期识别：变更对象 RFID、变更类型供应商变更、产品型号 DNBSEQ-G99、影响层级物料/子系统/系统级、风险点包括装配、初始化、读取、写入、EMC。

### TC-REQ-002：归口包匹配

系统应匹配 RFID 供应商变更验证包，并推荐包内测试条目。

### TC-REQ-003：条件触发项确认

安规 EMC 测试应要求用户确认是否纳入，并记录用户决策。

### TC-REQ-004：用户审核推荐条目

用户应能采纳、修改、删除、补充、合并和拆分推荐条目。

## 8. 验证方案生成测试

### TC-PLAN-001：生成验证方案草稿

系统应生成文档履历、概述、DUT 描述、参考文档、测试项目列表和每个 `3.x` 测试项目详情。

### TC-PLAN-002：测试项目详情生成

每个测试项目应生成测试目的/测试标准、测试方法/原理、测试工具、测试步骤、测试连接图或照片、测试记录、需求符合性和 BUG 信息。

### TC-PLAN-003：完整性校验

DUT 测试数量为空时，系统应阻断导出并提示补充。

### TC-PLAN-004：Word 导出

校验通过后，系统应导出可打开的 `.docx` 文件，章节结构符合模板。

## 9. 权限与隔离测试

| 用例 | 预期 |
|---|---|
| 项目隔离 | 用户只能查看授权项目的资料、需求和方案 |
| 资料标签范围过滤 | 项目只检索符合资料范围规则的资料 |
| 管理员审核权限 | 普通用户不能发布资料或归口包 |

## 10. 联网大模型测试

| 用例 | 预期 |
|---|---|
| 联网默认关闭 | 系统只使用本地知识库 |
| 联网参考开启 | 外部参考被标记，用户审核后才能进入方案 |
| 脱敏规则 | 内部编号、供应商敏感信息和未公开参数在联网前脱敏 |

|---|---:|

| 高度相似资料召回率 | ≥80% |

| 指标 | 目标 |
|---|---:|

| 测试标准提取准确率 | ≥80% |
| 测试步骤提取准确率 | ≥80% |
| 测试记录表提取可用率 | ≥75% |

### 风险解析

| 指标 | 目标 |
|---|---:|
| DFMEA 有效行识别率 | ≥85% |
| DFMEA 关键字段提取准确率 | ≥80% |
| Jira 问题字段提取准确率 | ≥80% |
| 风险补充建议采纳率 | ≥50% |

### 测试推荐

| 指标 | 目标 |
|---|---:|
| 变更对象识别准确率 | ≥85% |
| 变更类型识别准确率 | ≥85% |
| 归口包匹配准确率 | ≥75% |

| 必测项遗漏率 | ≤15% |

| 指标 | 目标 |
|---|---:|
| Word 结构符合率 | ≥90% |
| 测试项目列表完整率 | ≥90% |

```text
端到端主流程成功
核心指标达到最低目标
```
