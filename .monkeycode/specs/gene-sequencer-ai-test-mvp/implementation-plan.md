# MVP 实施计划与里程碑

## 1. 交付目标

MVP 需要在内部试点中完成以下闭环：

```text
资料上传
→ 标签识别与补充
→ 去重判断
→ 管理员审核发布
→ 验证方案拆分
→ 测试条目资产和归口包生成
→ Jira/DFMEA 风险解析
→ 新需求分析
→ 测试条目推荐与审核
→ 验证方案生成
→ Word 导出
```

## 2. 总体周期

推荐 10-12 周完成 MVP。

| 里程碑 | 周期 | 目标 |
|---|---:|---|
| M0 | 第 1 周 | 项目启动与设计确认 |
| M1 | 第 2-3 周 | 基础平台与资料池 |
| M2 | 第 4-5 周 | 文档解析、标签识别、资料审核 |
| M3 | 第 6-7 周 | 测试条目资产、归口包、风险知识源 |
| M4 | 第 8-9 周 | 新需求分析与测试推荐 |
| M5 | 第 10-11 周 | 验证方案生成与 Word 导出 |
| M6 | 第 12 周 | 试点验证、优化和上线准备 |

## 3. M0：项目启动与设计确认

交付物：MVP 需求规格、子系统目录 V1、验证方案模板 V1、数据模型草案、Prompt/Schema 草案、样例资料集。

验收标准：

| 编号 | 标准 |
|---|---|
| M0-A1 | MVP 功能范围确认 |
| M0-A2 | 验证方案模板可作为 docx 渲染模板 |
| M0-A3 | 样例资料可用于开发和测试 |
| M0-A4 | 技术栈和部署方式确认 |

## 4. M1：基础平台与资料池

开发内容：登录、项目列表、资料池列表、资料上传、用户角色、项目权限、MinIO 文件存储、Celery 任务框架。

验收标准：

| 编号 | 标准 |
|---|---|
| M1-A1 | 用户可登录系统 |
| M1-A2 | 用户可进入项目工作台 |
| M1-A3 | 用户可上传 Word/PDF/Excel |
| M1-A4 | 系统保存文件和文档元数据 |
| M1-A5 | 管理员可查看所有资料 |

## 5. M2：文档解析、标签识别、资料审核

开发内容：Word/PDF/Excel 解析、标签识别、标签补充、去重检测、管理员审核、文档切片和向量化。

验收标准：

| 编号 | 标准 |
|---|---|
| M2-A1 | 上传资料后系统自动生成标签建议 |
| M2-A2 | 用户可确认或修改标签 |
| M2-A3 | 系统能发现完全重复资料 |
| M2-A4 | 管理员审核通过后资料状态为已发布 |
| M2-A5 | 已发布资料完成切片和向量化 |

## 6. M3：测试资产与风险知识源

开发内容：验证方案拆分、测试条目分类映射、测试条目资产库、测试归口包、Jira 解析、DFMEA 解析、风险项关联测试条目。

验收标准：

| 编号 | 标准 |
|---|---|
| M3-A1 | 示例 RFID 验证方案能拆出 5 个测试条目 |
| M3-A2 | 每个条目能映射测试对象、层级、类型、子系统 |
| M3-A3 | 系统能生成 RFID 供应商变更验证包 |
| M3-A4 | Jira 表单能解析为风险项 |
| M3-A5 | DFMEA 能解析为风险项 |
| M3-A6 | 风险项能关联测试条目或归口包 |

## 7. M4：新需求分析与测试推荐

开发内容：需求输入、需求解析、归口匹配、风险补充、本地资料检索、联网参考、推荐清单、用户审核、条件触发确认。

验收标准：

| 编号 | 标准 |
|---|---|
| M4-A1 | 输入 RFID 供应商变更需求后识别对象和变更类型 |
| M4-A2 | 系统能匹配 RFID 供应商变更验证包 |
| M4-A3 | 系统能推荐装配、初始化、读取、写入、EMC 测试 |
| M4-A4 | DFMEA/Jira 能补充风险驱动测试建议 |
| M4-A5 | 用户能审核并形成最终测试条目清单 |
| M4-A6 | 每条推荐显示推荐理由和依据来源 |

## 8. M5：验证方案生成与 Word 导出

开发内容：方案草稿生成、概述生成、DUT 表、参考文档、测试项目列表、3.x 详情、在线编辑、完整性校验、Word 导出、版本记录。

验收标准：

| 编号 | 标准 |
|---|---|
| M5-A1 | 审核后的测试条目能生成测试项目列表 |
| M5-A2 | 每个测试条目生成 3.x 详情 |
| M5-A3 | 参考文档自动汇总 |
| M5-A4 | DFMEA 编号进入测试项目列表和需求符合性表 |
| M5-A5 | Jira 问题可插入 BUG 信息候选 |
| M5-A6 | 完整性校验能发现必填缺失 |
| M5-A7 | 导出的 Word 结构符合示例模板 |

## 9. M6：试点验证和上线准备

试点任务：导入 20-50 份真实资料、拆分至少 5 份历史验证方案、导入 Jira 和 DFMEA 样例、测试 3-5 条真实需求、组织测试工程师评审。

验收指标：

| 指标 | 目标 |
|---|---:|
| 标签识别可用率 | ≥80% |
| 验证方案拆分准确率 | ≥85% |
| 子系统映射可接受率 | ≥75% |
| 推荐测试条目采纳率 | ≥60% |
| Word 模板格式符合率 | ≥90% |
| 每条推荐有来源依据 | ≥90% |

## 10. 人员配置

推荐配置：产品/业务负责人 1 人、测试领域专家 1-2 人、后端工程师 2 人、前端工程师 1-2 人、AI/RAG 工程师 1 人、测试工程师 1 人、DevOps 0.5 人。

最小配置：1 名产品/领域专家、2 名全栈工程师、1 名 AI 工程师、1 名测试工程师。

## 11. 主要风险与对策

| 风险 | 对策 |
|---|---|
| 文档格式差异大 | 先支持验证方案 V1、DFMEA/Jira 样例格式 |
| 标签识别不稳定 | 规则 + LLM + 用户确认 |
| 测试条目拆分错误 | 拆分结果必须人工确认 |
| 归口包泛化过度 | 管理员审核后发布 |
| 推荐条目质量不稳定 | 归口包优先，AI 补充兜底 |
| Word 导出格式复杂 | 先固定一个模板 |
| 联网内容不受控 | 默认关闭，脱敏和外部参考标识 |
