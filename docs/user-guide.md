# 基因测序仪 AI 测试方案 MVP 使用说明

## 1. 登录与项目选择

系统提供两个试用账号：

| 角色 | 用户名 | 密码 | 权限 |
| --- | --- | --- | --- |
| 管理员 | `admin` | `admin123` | 上传资料、确认标签、发布资料、发布测试资产、生成方案 |
| 测试工程师 | `tester` | `tester123` | 上传资料、确认标签、发起需求分析、生成方案草稿 |

登录后在左侧菜单上方选择当前项目。项目用于限定资料检索范围、测试资产推荐范围和验证方案生成上下文。

创建项目空间：

1. 进入 `项目空间`。
2. 填写项目编码、项目名称和项目说明，例如 `G99 ECR4.0`。
3. 点击 `创建项目空间`。
4. 在顶部项目选择器或项目空间列表中切换到新项目。

删除项目空间：

1. 进入 `项目空间`。
2. 在项目列表中点击 `删除`。
3. 输入当前账号密码完成二次确认。

`项目工作台` 会展示当前项目的已发布资料数、测试条目数、风险项数、验证方案数、项目状态、推荐下一步和快捷入口。

## 2. 自由应用

自由应用用于基于当前项目资料库和已配置大模型自由提问。

操作流程：

1. 进入 `自由应用`。
2. 输入问题，例如 `G99 ECR4.0 需要重点关注哪些历史风险和回归测试？`。
3. 选择是否使用当前项目资料库。
4. 选择是否调用已配置大模型。
5. 点击 `提交问题`。

系统会优先检索当前项目的资料切片、测试条目、测试归口包和风险知识源。模型已配置时会生成综合回答；模型不可用时返回本地资料命中摘要。

## 3. 统一资料池

统一资料池是资料上传的唯一入口。测试规范、验证方案、测试报告、Jira 导出和 DFMEA 文件都从这里上传。

推荐操作流程：

1. 进入 `统一资料池`。
2. 点击 `上传资料`，选择文件。
3. 系统根据文件名和内容给出标签建议，例如产品型号、子系统、文档类型、变更类型。
4. 点击 `确认标签`，补齐或修正文档标签。
5. 管理员点击 `发布`。
6. 系统根据文档类型自动沉淀测试资产。

自动处理规则：

| 文档类型 | 自动处理结果 |
| --- | --- |
| 验证方案 | 解析切片，拆分测试条目，生成或更新测试归口包 |
| 测试规范 | 解析切片，拆分测试条目，生成或更新测试归口包 |
| 测试报告 | 解析切片，拆分测试条目 |
| Jira 导出 | 解析历史问题，生成风险知识项 |
| DFMEA | 解析失效模式、原因、控制措施和风险等级，生成风险知识项 |

## 4. 测试资产

测试资产页用于查看和审核系统自动生成的资产。

主要资产包括：

| 资产 | 来源 | 用途 |
| --- | --- | --- |
| 测试条目 | 已发布的验证方案、测试规范、测试报告 | 复用历史测试项目，支撑需求分析推荐 |
| 测试归口包 | 测试条目按对象、变更类型和子系统归并 | 在需求分析中推荐必测、建议和条件触发测试 |
| 风险知识源 | 已发布的 Jira 导出和 DFMEA | 补充历史问题、失效模式和回归风险 |

正常流程下无需手动输入资料 ID 或粘贴 CSV。`维护工具` 用于数据补救，例如历史资料迁移后需要重新拆分条目、补生成 RFID 供应商变更归口包、补解析风险源。

## 5. AI 大模型接入和调用

MVP 采用本地知识优先的 AI 编排方式。系统先使用统一资料池、文档切片、测试条目、测试归口包和风险知识源生成本地候选结果，再调用模型补全结构化 JSON。模型调用失败、超时或输出不符合 Schema 时，系统自动使用本地规则结果。

当前已接入的 AI 调用点：

| 调用点 | 入口 | 作用 |
| --- | --- | --- |
| 资料标签识别 | 统一资料池的 `AI 标签` | 基于文件名、已有标签、标签建议和文档切片补全产品型号、子系统、文档类型和变更类型 |
| 测试条目拆分 | 资料发布后自动执行，或测试资产维护工具补救执行 | 基于文档切片和本地拆分候选补全测试对象、层级、类型、步骤、工具和证据 |
| 风险解析 | Jira 导出或 DFMEA 发布后自动执行，或测试资产维护工具补救执行 | 基于 CSV/文本解析结果补全失效模式、后果、根因、控制措施、RPN 和建议测试 |
| 需求字段解析 | 需求分析页面 | 从需求文本中抽取测试对象、变更类型、产品型号、子系统和缺失字段 |
| 需求推荐 | 需求分析页面 | 基于本地归口包、风险项和知识检索命中补充必测、建议和条件触发测试 |
| 验证方案检查 | 验证方案页面 | 基于方案 JSON 和本地检查结果补充阻断项、警告和改进建议 |

模型接入采用 OpenAI 兼容接口。未配置模型时，系统使用本地规则兜底；配置模型后，各流程按上表调用模型补全结构化结果。

用户配置流程：

1. 使用任意账号登录。
2. 进入 `系统设置`。
3. 在 `AI 模型配置` 中选择 `OpenAI 兼容接口`。
4. 填写 Base URL、模型名称、API Key 和调用超时时间。
5. 点击 `保存设置`，确认 `AI 模型状态` 显示 `已接入`。

页面保存的是当前后端进程的运行时配置。服务重启后的默认值仍来自环境变量，生产部署建议同步更新 `.env.production` 或容器环境变量。

AI 输出约束：

| 约束 | 说明 |
| --- | --- |
| JSON 输出 | 模型只输出 JSON，系统按任务 Schema 校验字段 |
| 本地来源 | 需求推荐必须引用本地 `source_type`、`source_id` 和 `evidence` |
| 人工审核 | 标签、测试资产、推荐项和验证方案仍由测试工程师或管理员确认 |
| 自动回退 | 模型未配置、超时、异常或输出非法时使用本地规则结果 |

环境变量：

```bash
# 模型提供方，local 表示本地规则兜底，openai-compatible 表示启用 OpenAI 兼容接口
AI_PROVIDER="openai-compatible"

# OpenAI 兼容接口地址，例如内网模型网关
AI_BASE_URL="https://your-model-gateway.example.com/v1"

# 模型 API Key
AI_API_KEY="CHANGE_ME"

# 模型名称
AI_MODEL="your-model-name"

# 调用超时时间
AI_TIMEOUT_SECONDS="20"
```

系统接口：

```bash
# 查看模型接入状态
GET /api/ai/config

# 校验 AI 结构化输出
POST /api/ai/validate

# 记录可追溯 AI 调用
POST /api/ai/runs
```

外部联网大模型默认关闭。管理员开启后，外部模型仅作为受控参考来源，进入验证方案前需要脱敏、标记来源并经过人工审核。

## 6. 需求分析

需求分析用于把新需求或变更描述转成测试推荐。

单条需求操作流程：

1. 进入 `需求分析`。
2. 按标准需求格式输入需求。
3. 点击 `开始分析`。
4. 系统解析测试对象、变更类型、产品型号、子系统和缺失信息。
5. 系统根据测试归口包和风险知识源推荐测试项。

批量需求操作流程：

1. 进入 `需求分析`。
2. 点击 `下载需求批量分析模版`，获取 CSV 模板。
3. 使用 Excel 或兼容工具打开模板并填写多条需求。
4. 点击 `上传需求批量分析` 选择已填写的模板表格。
5. 点击 `开始分析`。
6. 系统逐行校验必填项，完整行自动分析，缺失行返回缺失字段。

需求模板字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| 需求标题 | 必填 | 用于区分每条需求的短标题 |
| 产品型号 | 必填 | 适用产品型号，例如 DNBSEQ-G99 |
| 变更对象 | 必填 | 本次变更涉及的对象，例如 RFID |
| 变更背景 | 必填 | 说明为什么需要这次变更 |
| 变更内容 | 必填 | 说明具体变化点 |
| 所属子系统 | 可选 | 对象所属子系统，例如 RFID；系统可根据变更对象推断 |
| 变更类型 | 可选 | 供应商变更、设计变更、工艺变更等；系统可根据背景和内容推断 |
| 影响范围 | 可选 | 说明可能影响的测试范围 |
| 验收标准 | 可选 | 说明通过标准 |
| 已知风险 | 可选 | 说明已知问题或风险点 |

推荐结果中会标明来源和依据，便于测试工程师审核。

## 7. 验证方案

验证方案页面用于把需求分析结果转换为验证方案草稿，并导出 Word 文件。

操作流程：

1. 在 `需求分析` 页面完成一次分析。
2. 进入 `验证方案`。
3. 使用需求分析 ID 生成方案草稿。
4. 点击 `完整性检查`，确认方案是否覆盖 DUT、参考文档、测试项目和风险项。
5. 点击 `导出 Word`，生成 `.docx` 文件。
6. 下载文件后按企业流程进行人工审查和签核。

导出的 Word 模板遵循示例验证方案结构：文档履历、概述、DUT 描述、参考文档、测试项目列表和测试项目详情。

## 8. 推荐使用路径

1. 管理员上传并发布历史测试规范、验证方案、测试报告、Jira 导出和 DFMEA。
2. 在 `测试资产` 页面审核并发布自动生成的测试条目和归口包。
3. 测试工程师在 `需求分析` 页面输入新需求或变更。
4. 系统推荐测试条目和风险补充项。
5. 测试工程师生成验证方案草稿。
6. 管理员或测试负责人执行完整性检查、Word 导出和人工审批。
